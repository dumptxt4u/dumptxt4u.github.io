14.1 Transaction Concept

What is a Transaction?

  • A unit of program execution
  • Accesses and possibly updates various data items

Initiation of a Transaction

  • Initiated by a user program
  • Written in high-level language (e.g., SQL)
  • Or programming language (C++, Java) with JDBC/ODBC

Delimiting a Transaction

  • begin transaction
  • end transaction

Operations in a Transaction

All operations executed between begin and end transaction

Appears as Indivisible Unit

Must appear to the user as a single, indivisible unit

All or None Execution

  • Executes in entirety
  • Or not at all

If Transaction Fails

Any changes must be undone

Reasons for Failure

  • Transaction error (e.g., divide by zero)
  • OS crash
  • Hardware failure

Challenge in Ensuring Atomicity

Changes in memory vs. on disk

Atomicity

“All-or-none” property

Isolation Introduction

Actions not separated by other operations

User Impression vs. Reality

Single SQL involves many accesses

Database System Actions

Ensure no interference from concurrent statements

Isolation

Transactions operate properly without interference

Durability Introduction

Actions persist across crashes

Durability

Changes persist even if system crashes

Transactions as Ideal Structure

Due to atomicity, isolation, durability

Consistency Requirement

Preserve database consistency

14.2 A Simple Transaction Model

Why a Simple Model?

SQL is complex; focus on data movement

Ignored Operations

Restricted to Arithmetic

Only arithmetic operations

Data Items

Single value, identified by name (A, B, etc.)

Bank Application Example

Accounts and updating transactions

read(X) Operation

Transfers from DB to main memory variable

write(X) Operation

Transfers from memory to DB

Change Visibility

Assumption on Write

write(A);
